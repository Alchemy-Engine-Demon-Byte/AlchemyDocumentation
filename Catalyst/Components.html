Components

Apr 04, 202612 min read

  • system/catalyst
  • type/reference

A component is a pure data structure tied to an entity. In the ECP architecture, components should ideally contain no logic, acting only as state containers for Processors to transform.

Declaration

To define a component, create a standard C++ struct, use the [[Reflect]] attribute, and the CATALYST_COMPONENT macro. This ensures the component is recognized by the reflection system and assigned a unique ID.

Usage in Code

Components are managed by the EntityLibrary. You can add, retrieve, or remove them using templated methods.

Component Slots

Component Slots allow a single entity to have multiple instances of the same component type, each identified by a unique slot name. This is essential for scenarios like NPCs with multiple mesh components (body, weapon, armor) that need to remain cache-efficient.

How Slots Work Internally

Slots are implemented as virtual TypeInfo entries. Each slot name is hashed into a unique TypeInfo that shares the memory layout of the base type but gets its own ID. From the archetype’s perspective, MeshComponent_Body, MeshComponent_Weapon, and MeshComponent_Armor are three entirely separate component types — each occupying its own column.

Unslotted vs Slotted Archetype Layout

Unslotted — an entity with TransformComponent and HealthComponent:

TypeRegistry
  "TransformComponent"  → ID: 0
  "HealthComponent"     → ID: 1

Archetype  Bitset: 1 1 0 0 0 ...
                   │ │
                   │ └─ HealthComponent (ID 1)
                   └─── TransformComponent (ID 0)

Columns (Structure of Arrays):
  Col[0]  TransformComponent  │ E1 │ E2 │ E3 │ E4 │ ...
  Col[1]  HealthComponent     │ E1 │ E2 │ E3 │ E4 │ ...

Slotted — an entity with TransformComponent and three MeshComponent slots:

TypeRegistry
  "TransformComponent"       → ID: 0
  "MeshComponent"            → ID: 1  ← base type, never set on any archetype
  "MeshComponent_Body"       → ID: 2  ← virtual (same struct, unique ID)
  "MeshComponent_Weapon"     → ID: 3  ← virtual
  "MeshComponent_Armor"      → ID: 4  ← virtual

SlotRegistry
  MeshComponent (ID 1)  →  [ Body (ID 2), Weapon (ID 3), Armor (ID 4) ]

Archetype  Bitset: 1 0 1 1 1 0 ...
                   │   │ │ │
                   │   │ │ └─ MeshComponent_Armor  (ID 4)
                   │   │ └─── MeshComponent_Weapon (ID 3)
                   │   └───── MeshComponent_Body   (ID 2)
                   └───────── TransformComponent   (ID 0)
                   NOTE: MeshComponent base (ID 1) is never set

Columns (Structure of Arrays):
  Col[0]  TransformComponent      │ E1 │ E2 │ E3 │ ...
  Col[1]  MeshComponent_Body      │ E1 │ E2 │ E3 │ ...  ← same sizeof(MeshComponent)
  Col[2]  MeshComponent_Weapon    │ E1 │ E2 │ E3 │ ...  ← same sizeof(MeshComponent)
  Col[3]  MeshComponent_Armor     │ E1 │ E2 │ E3 │ ...  ← same sizeof(MeshComponent)

The virtual TypeInfos are purely an ID trick — each slot column stores raw MeshComponent data. The SlotRegistry is the only thing that knows they all belong to the same base type, which is what ComponentSpan<T> and query matching use to reassemble them.

Slot Names Drive Archetype Identity

Because each unique slot name produces a unique virtual TypeInfo ID, two entities with different slot name sets land in different archetypes, even if the slot count is identical:

This splits entities across three separate archetypes, fragmenting memory and reducing ForEach iteration efficiency. Standardise slot names across entity types to keep related entities in the same archetype:

Use a consistent vocabulary of slot names (e.g. "Body", "Weapon", "Offhand") or a standardized namespace across all NPC types. Entities that genuinely differ in composition (a quadruped with four leg meshes vs. a biped) belong in separate archetypes anyway — that fragmentation is expected and correct.

Slot Hierarchies

Slots on a single entity can declare parent-child relationships using the slot hierarchy API. This is useful for expressing that a sword mesh belongs under a body mesh, or that a collider follows a specific mesh slot — without requiring any new component types or archetype changes.

Hierarchy data is stored in an invisible internal SlotHierarchyComponent that is automatically added to the entity on first use. You never interact with it directly.

Declaring Hierarchies on Prefabs

Slot hierarchies can be declared directly on an EntityPrefab so that every spawned entity already has the correct parent relationships — with no archetype migration cost at spawn time.

The prefab API mirrors the runtime API exactly:

Prefab methodRuntime equivalent
Prefab.SetSlotParent("Child", "Parent")Library.SetSlotParent(Entity, "Child", "Parent")
Prefab.ClearSlotParent("Child")Library.ClearSlotParent(Entity, "Child")
Prefab.HasSlotParent("Child")Library.HasSlotParent(Entity, "Child")

Key Properties

  • Cross-type parenting is allowed. A ColliderComponent slot can be the child of a MeshComponent slot. The hierarchy stores slot names only — types are only used at the query level for the return value.
  • The SlotHierarchyComponent is invisible. It is never returned by general-purpose queries. Think of it as metadata attached to the entity, not a game-facing component.
  • No archetype change at runtime. Setting or clearing a parent relationship only modifies a std::unordered_map inside SlotHierarchyComponent. The entity does not migrate archetypes. When declared on a prefab, SlotHierarchyComponent is included in the baked archetype upfront, so spawn cost is zero.
  • Slots are unique per entity. Because slot names are unique across all component types on a single entity, no type disambiguation is needed in SetSlotParent or ClearSlotParent.
Entity: "Warrior"

  MeshComponent["Body"]           ← root (no parent)
  ├── MeshComponent["Sword"]      ← child of "Body"
  │   └── ColliderComponent["SwordCollider"]  ← child of "Sword"
  └── MeshComponent["Shield"]     ← child of "Body"
      └── ColliderComponent["ShieldCollider"] ← child of "Shield"

Auto-Generated Slot Names

When a [[Slottable]] component is added without an explicit slot name, the engine automatically generates a name like "ComponentType_0", "ComponentType_1", etc. This provides a safety net for code that doesn’t explicitly track slot names.

Best Practices: Always Use Explicit Slot Names

Do not rely on auto-generated slot names. While they provide a safety net, they make code brittle and hard to understand. Always assign meaningful, explicit names to slots:

Why explicit names matter:

  • Readability: Anyone reading GetComponent(entity, "Weapon") immediately understands what’s being retrieved
  • Stability: If you refactor the order of additions, auto-generated indices shift; named slots never do
  • Debugging: Named slots appear in profilers and logs with semantic meaning
  • Intent: Your code explicitly states “this is the weapon mesh”, not “this is the second mesh”

Auto-generated names exist only as a fallback for simple prototypes or one-off code. Production code should always use explicit names.

Toggle Components

Components can be disabled and re-enabled at runtime without destroying them. A disabled component’s data is fully preserved in memory — it is simply invisible to queries, so processors that iterate that component type skip the entity entirely with zero per-iteration overhead.

This is the correct tool for world streaming, LOD transitions, and any scenario where entities should temporarily stop participating in simulation without the cost of destruction and reconstruction.

How It Works

Disabling a component performs a single archetype migration. The entity moves from an archetype containing T into one containing an internal __Disabled__T virtual type. Because the query’s cached archetype list never includes that destination archetype, disabled entities are never discovered — there is no per-entity branch or check.

Before DisableComponent<MeshComponent>(E):
  Archetype [Transform | MeshComponent]   ← query sees this

After:
  Archetype [Transform | __Disabled__MeshComponent]  ← query does NOT see this

Re-enabling performs the reverse migration and restores the original data exactly.

Basic API

DisableComponent is a no-op if the entity does not have the component or if it is already disabled.

Performance

Because disabled entities live in a completely separate archetype, query iteration scales linearly with the number of active entities only. Disabling 50% of entities reduces ForEach time by approximately 50% — there is no scan, skip, or branch for disabled entities.

The migration cost on DisableComponent / EnableComponent is the same as a regular AddComponent / RemoveComponent archetype transition. This is a boundary event, not a per-frame cost.

Constructors

Catalyst supports Constructor Annotations. This allows you to run specific initialization logic whenever a component is added to an entity, which is useful for registering data with external systems or generating unique IDs.

These constructors are executed:

  1. Immediately after AddComponent.
  2. During EntityTransaction::Commit.
  3. During EntityPrefab::Instantiate.

Architectural Details (Advanced)

Memory Layout (SoA)

Components are stored in Archetypes using a Structure of Arrays (SoA) format. All components of the same type for a specific archetype are stored contiguously in memory.

Pointer Stability

Catalyst uses Chunked Memory (16KB Pages) for component storage. This ensures Pointer Stability: a pointer returned by GetComponent will remain valid for the lifetime of that component on that entity, even if millions of other entities are spawned or other archetypes reallocate.

Bitset Identification

Each component type is assigned a unique uint32_t ID by the TypeRegistry. An entity’s composition is represented by a ComponentBitset, allowing for O(1) comparisons and extremely fast query matching.
