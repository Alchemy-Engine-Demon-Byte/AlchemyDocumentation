Processors are the logic units of the Alchemy ECS. While the Entity Library holds the data, Processors define how that data transforms over time. They are managed by the EntityLibrarySubsystem and executed automatically every frame.

Creating a Processor

To create logic, inherit from IProcessor and implement Execute. The most common pattern is to iterate over entities using a EntityQuery.

Configuration & Ordering

Processors often depend on each other (e.g., Physics must run before Rendering). You configure this in the constructor using Tags.

Thread Affinity

By default, Processors run on background worker threads (JobAffinity::Any). If you need to access main-thread-only APIs (like Windowing or OpenGL), you must specify it.

Inline Processors

For cases where a full subclass is unnecessary — editor tooling, debug hooks, test fixtures, or short-lived runtime logic — use InlineProcessorBuilder to create a processor directly from lambdas.

Both callbacks receive an IProcessor* pointing to the processor itself. All configuration — ExecuteBefore, ExecuteAfter, ExecutionGroup, LibraryFilter — is set directly on that pointer inside OnInitialize. This means the builder never has to mirror IProcessor’s fields; any new fields added to IProcessor are automatically available with no changes to the builder.

When to Use a Named Processor Instead

Inline processors are a convenience, not a replacement. Prefer a named subclass when:

  • The processor will be reused across multiple libraries or modules
  • It carries significant state (many query IDs, configuration fields, caches)
  • It needs to be auto-registered via [[Reflect]] so it applies to all matching libraries automatically

Advanced Usage

The Active JobBuilder

During the execution of a Processor, Catalyst maintains a thread_local pointer to the active JobBuilder. This allows helper functions like ForEachParallel to automatically attach their sub-tasks to the current frame’s execution graph without explicit parameter passing.

Manual Job Dispatch

Sometimes you need raw control over the Job System. You can access the JobBuilder directly from the ProcessorContext to dispatch custom tasks.

Entity Command Buffers (ECB)

Processors are physically prohibited from making immediate structural changes (adding/removing components or destroying entities) to the EntityLibrary while executing. This ensures:

  1. Thread Safety: Multiple processors can run in parallel without race conditions on memory layout.
  2. Iterator Stability: Adding an entity won’t cause a query iteration to skip or repeat entities due to archetype transitions.

Instead, all structural changes are recorded into the EntityCommandBuffer provided by the ProcessorContext.

Core Commands

The ECB supports all standard ECS operations. These are recorded in O(1) time and played back at the end of the frame.

Toggle Commands

Toggle Components must also go through the ECB inside processors, since enabling or disabling a component is a structural change (archetype migration).

The toggle direction for ToggleComponent is resolved when the command buffer plays back, not when you record it. This means the flip always reflects the entity’s actual state at the end of the frame rather than at the moment the processor ran.

Deferring Custom Logic

Beyond structural changes, you can defer arbitrary code execution using Defer. This is useful for logic that must run on the main thread or must happen after the world state has stabilized.

Note on Captures: When using Defer, always capture by value ([=]). Capturing by reference is unsafe as the processor’s stack frame will be gone by the time the command buffer plays back.

Implementation Details

  • Linear Allocation: The ECB uses a page-based linear allocator (16KB pages). Recording a command is essentially a pointer increment, making it extremely fast.
  • Determinism: Commands are played back in the exact order they were recorded. If System A records a change before System B, that order is preserved during playback. - We have made the decision to sacrifice performance over determinism when it comes to the ECB. There are ways of batching commands or sorting them for better cache efficiency, or maybe even using the job system to multithread these commands, but in nearly every case when you’re working with the command buffer, you want it to be deterministic and on the game thread.
  • Playback Window: The EntityLibrarySubsystem automatically flushes the command buffer once all processors have reached a synchronization point (usually the end of the Update tick).
