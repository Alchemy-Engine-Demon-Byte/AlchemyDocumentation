Alchemy Asset Manager

The Asset Manager is the engine’s central hub for loading, saving, and managing binary data. It provides a high-performance, version-aware serialization system that ensures data integrity and type safety across all engine modules.

Quick Start: The Basics

At its simplest, the Asset Manager allows you to load and save EntityLibrary objects (which act as containers for your game data) and most other types of assets, such as textures.

Creating Reflectable Assets

To make a custom C++ struct saveable as an asset, simply mark it with the [[Reflect]] attribute. The Alchemy Reflector will automatically generate the necessary binary serialization code.

Note: All fields are serialized by default. Use [[Transient]] to skip fields that shouldn’t be saved (like runtime-only timers or pointers).

Technical Architecture: Under the Hood

As you move deeper into the engine, the Asset Manager provides “Industry Grade” features to prevent crashes and data corruption.

1. The Asset Header

Every .alchemyasset file begins with a standardized header:

FieldPurpose
MagicA 4-byte signature (ALCH) to verify the file is a valid Alchemy asset.
Engine VersionThe Major.Minor.Patch version of the engine that saved the file.
Type HashA unique ID for the data type (e.g., EntityLibrary vs. Texture).
Asset VersionA custom version number for that specific asset’s format.

2. Type Safety via Compile-Time Hashing

To prevent you from accidentally loading a Texture file into an Animation struct, the engine uses GetTypeHash<T>().

This is a consteval function that generates a unique 32-bit ID based on the compiler’s function signature. If the TypeHash in the file header doesn’t match the TypeHash of the class you are trying to load, the Asset Manager will stop and log an error before any data is parsed.

3. Versioning & Backward Compatibility

The engine tracks two types of versions:

  • Engine Version: Global format changes (e.g., changing how Alchemy::String is stored).
  • Asset Version: Changes to a specific system (e.g., adding a new field to all EntityLibrary assets).

4. The Binary Archive

Unlike JSON or XML, Alchemy uses a custom binary Archive.

  • Speed: Data is bit-streamed directly from disk to memory.
  • POD Optimization: The Archive detects “Trivially Copyable” types (like int, float, or structs of primitives) and performs a single memcpy for the entire block rather than iterating field-by-field.
  • Mode Awareness: The same operator& logic is used for both Saving and Loading, reducing code duplication and human error.

Asynchronous Loading

For production environments where hundreds of assets must be loaded without hitching the game thread, the Asset Manager provides a non-blocking AsyncLoadAsset method.

Usage

Key Features of Async Loading:

  1. Job System Integration: Loading tasks (File IO + Deserialization) are dispatched to the JobSubsystem and run on background worker threads.
  2. Request Deduplication: If five different systems request the same asset at the same time, the Asset Manager will only perform one disk read. All five callbacks will be queued and executed once the single load operation completes.
  3. Thread Safety: While the data is loaded in the background, the completion callbacks are automatically dispatched back to the Main Thread, ensuring you can safely modify game state or spawn entities in response to an asset being loaded.

Batch Asynchronous Loading

When loading large sets of assets (e.g., streaming in a new world region), using individual AsyncLoadAsset calls in a loop can hitch the game thread due to job dispatch overhead. To solve this, the Asset Manager provides BatchAsyncLoadAssets.

Usage

Why Use Batching?

  1. Reduced Dispatch Overhead: Instead of hitting the job system 50 times for 50 assets, the batch system uses ParallelForBatch to submit the entire workload in a single “wave.” This keeps the game thread 100% smooth even when requesting hundreds of assets.
  2. The “Wave Strategy”: The system processes assets in small chunks (e.g., 5-10 at a time per worker thread). This staggers the OS-level file opening calls, preventing the “Thundering Herd” effect where too many simultaneous disk requests actually slow down the overall throughput.
  3. Instruction Locality: Because worker threads process multiple assets in a tight loop, the CPU stays in the “Deserialization” instruction path longer, improving cache efficiency.
