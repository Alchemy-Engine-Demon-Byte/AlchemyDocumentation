Sigil

Sigil is the UI framework for Alchemy Engine. It has two layers, used for different jobs:

  • Retained mode — a persistent UI built from Catalyst entities. Each widget is an entity; you build a tree once and mutate it through held handles or per-frame Render callbacks. This is the layer for game and application UI (HUDs, menus, panels). It is rendered by SigilRenderer.
  • Immediate mode (SigilIM) — a thin wrapper over ImGui for tools, debug overlays, and editor panels. See SigilIM.

This page covers the retained layer.

The model in one paragraph

A widget’s identity is its entity. There is no name registry and no per-frame reconciliation of the tree — every AddX() creates a fresh child entity and hands you back a Node (a thin EntityRef wrapper). You build the tree once, keep the handles you intend to touch, and update them later with setters or a Render callback. A single rule keeps this honest: a builder decides what exists; a Render callback or event handler decides what it currently shows. Anything that reads live game state belongs in a callback, rarely in the builder.

Registering a UI

Sigil renders any EntityLibrary you register with the SigilSubsystem. The build callback runs every frame during the build pass; gate it with a bool so the retained tree is constructed once.

For a UI authored as a reusable asset, derive from UIEntityPrefabAsset, implement Build(Canvas&), and spawn it with Sigil->SpawnView<MyView>(Library) — the returned root entity carries the whole tree and destroying it cascade-destroys every widget.

Building a tree

Canvas is the root context; Node is every widget. Layout nodes arrange their children; leaf nodes draw content.

Layout nodes (each takes a NodeProps): AddVBox, AddHBox, AddStack (children layered), AddCanvas (children positioned by their own anchor), and AddNode (a bare container that arranges nothing — the building block for a custom-rendered widget).

Leaf nodes: AddText(content), AddImage(texturePath), AddProgressBar(props).

NodeProps — every field optional: Size (a Vector2D; use UISize::Auto to shrink-wrap and UISize::Fill to consume remaining space), Anchor (UIAnchor), Padding (UIPadding), Gap (float, between children), BackgroundColor (Optional<Color>).

Input and events

Attaching any input callback makes the node hittable — the input handler hit-tests the cursor against the node’s laid-out rect. OnClick fires when a press and release both land on the node.

The thin void(Node&) overload is the common case and consumes the event automatically; the void(Node&, UIInputEvent&) overload hands you the event. Hover transitions are not consumable — they are state changes.

Dynamic child counts — the List node

A data-driven number of children is the one genuine place reconciliation is needed, so it lives in exactly one place: Node::List. Each frame the count is evaluated and children are spawned/despawned to match; a template builds each slot once, on creation.

Full details, including the serializable reflected-method form, in List Component.

Custom-rendered widgets

The built-in widgets (background/border, text, image, progress bar) are not special-cased in the renderer — each is a component with an EmitDraw method, dispatched through a per-node thunk list. A plugin adds a mesh-on-a-plane or a shader-driven widget the exact same way, with no engine change. See Custom Renderable Components.

How a frame flows

The SigilSubsystem tick runs, per registered library:

  1. Build — the registered callback (retained trees gate themselves to build once).
  2. List reconcile — spawn/despawn dynamic children to match their counts.
  3. Render callbacks — every node’s Render delegate fires.
  4. Layout — the flexbox-subset solver computes each node’s FinalRect (dirty subtrees only).

Then Render() walks the tree depth-first and records geometry into the Vulkan renderer in painter order (parent before children).
