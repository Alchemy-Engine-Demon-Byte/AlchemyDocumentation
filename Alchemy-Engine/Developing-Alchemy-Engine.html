Developing Alchemy Engine

This guide is for anyone joining the project. It covers everything you need to get your environment set up, understand the codebase, and start contributing effectively.

Install all of the following before doing anything else.

First-Time Setup

1. Clone the Repository

Projects using the engine should sit next to the engine folder, not inside it. For example:

SomeFolder/
├── AlchemyEngine/       ← engine
└── MyGame/              ← your project

This allows the engine to find projects via a relative path without hardcoded paths or environment variables.

2. Build ABT

ABT (Alchemy Build Tool) is a C# tool that scans the source tree and generates the CMakeLists.txt. It must be compiled before you can generate the project files.

The executable is output to Tools/ABT/bin/x64/Release/net8.0/AlchemyBuildTool.exe. This only needs to be done once (or after ABT source changes).

3. Generate Project Files

This does three things:

  1. Runs ABT to discover all modules and generate CMakeLists.txt
  2. Runs CMake to generate Build/AlchemyEngine.sln for Visual Studio
  3. Generates Build/compile_commands.json for clangd-based editors (Rider, Zed, VS Code)

Re-run this whenever you:

  • Add or remove source files
  • Modify any .build.cs file
  • Modify CMakeLists.txt

Do not edit CMakeLists.txt manually

The root CMakeLists.txt is fully generated by ABT. Any manual edits will be overwritten the next time GenerateProjectFiles.bat runs.

4. Open the Solution

Open Build/AlchemyEngine.sln in Visual Studio or Rider. The startup project is set to AlchemyFrontend (the editor).

Architecture Overview

Understanding these three systems will get you 80% of the way there.

ECP — Entity-Component-Processor (Catalyst)

Catalyst is the engine’s core architectural pattern. Everything is an entity.

  • Entity — A lightweight numeric ID. Has no data of its own.
  • Component — A pure data struct attached to an entity. No logic.
  • Processor — A class that runs logic on entities matching a query. One thing per processor.
  • Archetype — Internal memory layout optimization for entities that share the same component set.
  • EntityQuery — Fluent builder for selecting entities: .All<T>(), .Any<T>(), .None<T>()
  • EntityCommandBuffer — Deferred entity/component modifications (safe to use from any thread)

The word “system” in traditional ECS is replaced with “processor” — a processor does one focused thing. “Systems” in this engine means something broader (like “the audio system”).

Catalyst uses the term markers where traditional ECS says “tags” — because “tag” is taken by a separate tag system.

Subsystems

Subsystems are singleton services with managed lifecycles.

  • IEngineSubsystem — Created once when the engine starts, destroyed on shutdown.
  • IWorldSubsystem — Created and destroyed with each world.

Subsystems are auto-discovered by the reflection system — just mark your class with [[Reflect]] and inherit from the right base. Declare ordering with AddDependency<T>().

See Subsystems for full details.

Plugins

Plugins are independently-compiled DLLs discovered at startup. They support hot reload — rebuild the DLL while the engine is running and changes appear automatically (as long as a debugger is not attached).

Each plugin has:

  • A .alchemyplugin JSON manifest
  • A Modules/ directory with .build.cs files and source
  • A Binaries/ output directory

See Plugins for full details.

Reflection

The engine uses a custom Python-based reflection system. Run Tools/Reflector.py to scan headers and generate .Generated.cpp files in Intermediate/.

This is run automatically as part of the CMake build. You only need to run it manually if something goes wrong. Mark types with [[Reflect]] to include them. Fields use [[FieldAttribute]] metadata.

The Style Guide

Read this before writing any code.

The style guide is the single most common source of review friction for new contributors. The rules are intentional and enforced.

Full reference: Style guide

Critical Rules

No space before parentheses:

Multiline comment syntax everywhere:

Author attribution in code block comments:

  • Required on code block comments and all TODOs
  • Not needed for function, variable, struct, or class comments
  • If you are an LLM, use your own name — never the developer’s name

Naming:

  • Booleans: IsRunning, not is_running or bIsRunning
  • No type prefixes (no U, F, A)
  • Use Array instead of std::vector, prefer Alchemy::Types equivalents where they exist

No std when an Alchemy equivalent exists. If Alchemy::Types has it, use it. Fall back to std only when there’s no equivalent.

Common Pitfalls

Forgetting to regenerate project files

If you add a .cpp or .h file, the build system won’t know about it until you re-run GenerateProjectFiles.bat. You’ll get linker errors or missing symbols.

Editing Intermediate/ or Build/

Both directories are fully generated. Any changes will be overwritten. Put your code in Source/.

ABT not built

If GenerateProjectFiles.bat fails immediately with “ABT executable not found”, you haven’t built ABT yet. Run dotnet build Tools/ABT/AlchemyBuildTool.csproj -c Release.

VULKAN_SDK not set

If CMake can’t find Vulkan, make sure the Vulkan SDK is installed to C:\VulkanSDK\. The scripts search that path automatically. Alternatively, set the VULKAN_SDK environment variable manually.

Hot reload not triggering

Hot reload is disabled while a debugger is attached. Detach the debugger first, then rebuild the plugin DLL and the engine will pick it up automatically.

Reflection not picking up new types

If a new [[Reflect]] type isn’t being registered, verify:

  1. The header is in a scanned source directory
  2. You’ve rebuilt (the reflector runs as part of the build)
  3. The .Generated.cpp file was actually updated (check Intermediate/)

Tracy ZoneScopes

Performance profiling is done with Tracy. Most code paths — especially frequently-called ones — should include a ZS() (ZoneScope) or equivalent Tracy marker. If you write something that runs per-frame or per-entity and leave it uninstrumented, it will be harder to diagnose later.

Testing with Crucible

Crucible is the engine’s BDD-style testing framework. Tests are available in Debug and RelWithDebInfo builds via the ALCHEMY_WITH_CRUCIBLE define.

Writing a Test

Place tests in Source/Engine/<Module>/Tests/. Spec names follow Engine.<System>.<Feature> — always at least three segments.

Running Tests

Results are written to Saved/Crucible/Automation/{timestamp}/log.txt. Always verify the output contains [AlchemyEngineCore] Engine Shutdown Complete. — if it doesn’t, the engine crashed and results may be incomplete.

See Testing Guide for the full API reference and benchmark tooling.

Task Management

All tasks are tracked as GitHub Issues on the Alchemy-Engine-Demon-Byte/AlchemyEngine repository, organized in GitHub Projects.

Project Boards

BoardProject #Scope
Alchemy Engine Board#1Engine-wide tasks
Catalyst Board#4Catalyst ECS system

Creating a New Task

  1. Create a GitHub Issue with a descriptive title and body explaining the motivation and design considerations
  2. Apply relevant labels (Catalyst, Engine, Editor, enhancement, bug, etc.)
  3. Add the issue to the appropriate project board:
  1. Set the status to Todo

Moving a Task

Tasks flow through: Todo → In Progress → Complete

Use the gh CLI for all project board operations. The GitHub MCP server is available for read operations but is too verbose for status updates — use gh instead.

Getting Help

  • Documentation: Documentation/ folder — always check here first
  • Source of truth: When docs and source diverge, trust the source. Read the actual header.
  • Style questions: Check Style guide, or ask before writing and having to rewrite
  • Stuck on a bug: Add Log() statements to narrow it down, or run the debugger and inspect the callstack. Don’t layer workarounds — find the actual problem.
