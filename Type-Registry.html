Type Registry

The Type Registry is the central reflection database for the Alchemy Engine. It maintains runtime type information for every [[Reflect]]-marked struct and enum in the engine, enabling systems like Catalyst, serialization, config binding, and the editor to query type metadata without compile-time dependencies.

The registry lives in Source/Engine/Reflection/TypeRegistry.h and is part of the Alchemy namespace.

Overview

When a struct or enum is marked with the [[Reflect]] attribute, the Reflector (a build-time code generator) emits registration code that inserts a TypeInfo entry into the global TypeRegistry. Each entry carries:

  • Identity — name, owning module, and a unique numeric ID
  • Layout — size, field offsets, and per-field type names
  • Lifecycle — constructor, destructor, copy and move function pointers
  • Serialization — binary and config-text codec function pointers
  • Catalyst metadata — element indices, sparse storage flag, zero-size flag
  • Default instance — a byte snapshot of a default-constructed value

Systems query the registry by type (template), by name (string), by ID (uint32), or by std::type_index to resolve this metadata at runtime.

TypeInfo

Every registered type gets a TypeInfo struct. This is the primary data structure the registry stores and returns.

Registration

Types are registered at DLL load time through generated code emitted by the Reflector. There are four registration paths:

During registration the system:

  1. Creates a module-qualified key ("Engine.MyComponent")
  2. Allocates a unique ID
  3. Sets Size, IsZeroSizeElement, and StorageIsSparse via compile-time checks
  4. Assigns lifecycle function pointers based on SFINAE trait checks
  5. Snapshots a default-constructed instance into DefaultInstance
  6. Maps std::type_index(typeid(T)) to the TypeInfo pointer

The TypeToInfo mapping is updated every time RegisterStruct is called (even for duplicates), which ensures the pointer stays valid across DLL reloads.

RegisterSlotType(BaseType, SlotName)

Creates a virtual TypeInfo that clones an existing type’s layout and lifecycle but receives a unique ID and name. Used by the Catalyst slot system to give each (ComponentType, SlotID) pair its own archetype column.

RegisterVirtualType(Name, IsSparse)

Fabricates a standalone, zero-size TypeInfo with no fields or lifecycle functions. Used by Catalyst’s virtual (string-named) markers — the name is the fully qualified registry name (e.g. "$Marker$Player") and IsSparse fixes the storage policy for the type’s lifetime.

Lookup

The registry provides five lookup overloads:

GetType<T>()

Template-based lookup using std::type_index(typeid(T)). Fastest path — no string comparison. Returns nullptr if the type is not registered.

GetType(string_view Name)

String-based lookup with two-phase resolution:

  1. Exact key match — attempts a direct lookup in the Types map. Hits when passed a module-qualified name (e.g. "Engine.TransformComponent") or a bare name for hand-registered types
  2. Bare name fallback — if the exact key misses, strips any <Module>. prefix and searches the BareNameIndex. If multiple modules share the same bare name and no module filter is provided, a warning is logged and the first candidate is returned

GetType(uint32_t ID)

ID-based lookup via the IDToInfo array. O(1) access.

GetType(type_index Index)

Runtime type-index lookup. Used by Object to resolve its dynamic type via typeid(*this).

GetAllTypes()

Returns a const reference to the complete Types map. Used by the runtime module to iterate all registered types.

Instance Management

The Type Registry supports a dual-instance model to handle DLL boundaries:

Local Instance

Each DLL has its own local registry (a function-local static returned by GetLocalInstance()). During DLL load, static initializers and generated registration code populate this local instance.

Active Instance

A single static pointer (ActiveInstance) designates the “authoritative” registry. By default, Get() returns the local instance. Once the engine calls SetInstance(), all subsequent Get() calls return the active instance.

Merge Flow

When a DLL loads, the typical sequence is:

  1. Static initializers register types into the DLL’s local TypeRegistry
  2. OnLoad is called on the DLL module
  3. The module calls Merge() to copy all types from the local registry into the host registry
  4. SetInstance() redirects subsequent lookups to the host

This ensures all modules share a single type database after initialization, while each DLL can register its own types independently during load.

Plugin Context

The EditorContext struct (passed from host to guest DLLs) carries a TypeRegistryInstance field as void*. Plugins cast this back to TypeRegistry* to access the shared registry.

Usage in Engine Systems

The Type Registry is queried extensively throughout the engine:

  • Catalyst — Every component add/remove/query operation resolves TypeInfo* to determine element indices, storage policy, and archetype columns. Function-local static TypeInfo* caches are used to pay the lookup cost exactly once per type per binary lifetime
  • Object System — Object::LoadConfig and SaveConfig walk the inheritance chain by resolving BaseTypeName through the registry
  • Config Binding — ConfigArchive uses ConfigSerialize function pointers and per-field ConfigCodec thunks to serialize types to/from config text
  • Entity Library Serialization — Resolves component types by name when loading saved entity libraries
  • Editor Details Panel — Uses DefaultInstance byte snapshots to detect modified fields and support per-field reset
  • Instanced Struct — Validates that the requested type matches the stored TypeInfo before casting
  • Tag System — Resolves tag container types through the registry

Best Practices

  • Always use GetType<T>() (template) when the type is known at compile time — it avoids string lookups entirely
  • Cache TypeInfo* results in function-local static variables when calling repeatedly in hot paths
  • Use module-qualified names in GetType(string) when possible to avoid ambiguity warnings
  • Never call registration methods from multiple threads — registration happens at DLL load time on a single thread
  • The registry is not thread-safe during Merge() — ensure merges complete before spawning worker threads
