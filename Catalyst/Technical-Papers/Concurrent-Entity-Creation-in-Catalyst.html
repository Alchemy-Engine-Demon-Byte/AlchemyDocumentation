Concurrent Entity Creation in Catalyst

Who this is for. This paper assumes you know what an Entity-Component-System (ECS) is — entities are IDs, components are data, and systems iterate over them. It does not assume you know how Catalyst stores that data internally, and it does not assume you know terms like chunk, spine, paged store, or stable addressing. Those are introduced from scratch. By the end you should understand not just what we built, but why each decision was made and what we would do differently.

1. Abstract

Catalyst is the Alchemy Engine’s ECS. Historically, like most ECS implementations, it required that all entity creation happen on a single thread — the “game thread.” This paper describes the work that removed that restriction. Any thread may now create an entity and immediately read or modify it, with correctness guaranteed by lock-free and lock-based primitives rather than by deferring the work to a safe point later.

The headline results:

  • Single spawns are callable from any thread and, when two threads spawn into the same archetype, they now run genuinely in parallel.
  • Bulk spawns (world load, procedural generation, asset registry population) run ~2.3× faster than the original single-threaded path.
  • Reads outside the frame tick (ForEach called from editor/UI code) are safe to run concurrently with spawns, and pay zero extra cost when no spawn is in flight.

Getting there required redesigning how Catalyst stores entity data so that memory never moves out from under a concurrent reader or writer. It also produced a few surprises — most notably that the thing we assumed was the bottleneck (a lock) was not, and the real cost was hiding somewhere far less obvious.

2. Goals and non-goals

The work was scoped tightly up front.

Goals

  1. Any thread can create an entity and touch it immediately. Not “queue a request and get the entity next frame” — actually create it, get a handle back, and write to its components on the spot.
  2. Bulk creation must be fast. Loading a world or generating terrain can create hundreds of thousands to millions of entities at once. That path must exploit all cores.
  3. Prefabs and cross-library merges must work concurrently. These are the realistic ways entities get created in bulk, so they can’t be second-class.

Explicit non-goals

  • Concurrent destruction. Deleting entities from many threads at once is a much harder problem (it moves existing data around), and we decided it wasn’t worth the complexity yet. Where a system needs to “remove” an entity from multithreaded code, the intended pattern is entity disabling — flip a flag, and reap it later on the game thread. Concurrent destruction is discussed in Future work.

2.1 Why not just defer the work?

The obvious way to make entity creation “thread-safe” is to not actually do it on the calling thread. You record “please create this entity” into a per-thread command buffer, and a single thread plays all those commands back at a safe point. This is what an Entity Command Buffer (ECB) does, and Catalyst already has one.

We rejected ECB as the creation mechanism, deliberately. The whole point of the feature is “create it and touch it now.” If creation is deferred, you don’t have an entity yet — you have a promise of one — so you can’t read its components, can’t parent it to something, can’t hand its handle to another system this frame. Deferral solves a different problem (structural change during iteration) and would defeat the goal. So the rule became: creation is immediate; correctness comes from the data structures, not from postponing.

3. A primer on how Catalyst stores entities

To understand the concurrency problem, you first have to understand the storage. This section builds up the vocabulary. If you already know Catalyst’s internals, skim to §3.5.

3.1 Archetypes and columns

Catalyst groups entities by their exact set of components. Every unique set is an archetype. All entities that have exactly {Position, Velocity} live together in one archetype; entities with {Position, Velocity, Health} live in a different one.

Within an archetype, component data is stored column by column (a “structure of arrays” layout). One contiguous run of Position values, a separate run of Velocity values, and so on. Row N across all columns is entity N’s data. This layout is what makes iteration fast: a system that reads Position walks one tight, cache-friendly array.

graph TD
    subgraph Archetype["Archetype {Position, Velocity}"]
        P["Position column<br/>[P0][P1][P2][P3]..."]
        V["Velocity column<br/>[V0][V1][V2][V3]..."]
        H["EntityHandles<br/>[H0][H1][H2][H3]..."]
    end
    note["Row 2 = entity H2's Position P2 + Velocity V2"]

Each archetype also keeps a parallel EntityHandles array: row N holds the handle of the entity occupying that row. This is how iteration knows which entity it’s looking at.

3.2 Chunks

A column isn’t one giant allocation. It’s split into fixed-size blocks called chunks — in Catalyst, 16 KB each. A chunk holds as many rows as fit: for a 40-byte component, that’s ~400 rows per chunk. When a column fills its current chunks, it allocates another 16 KB chunk.

Chunks exist for cache and allocation reasons: 16 KB is a friendly size to allocate and to stream through the CPU cache, and you never have to copy a huge array to grow it — you just add a chunk.

3.3 The spine

If the data lives in chunks, something has to remember where the chunks are. That “list of pointers to chunks” is what we’ll call the spine. Think of it as the backbone of a column: the chunks are the ribs hanging off it.

spine:   [ ptr ][ ptr ][ ptr ]         <- an array of pointers
            |      |      |
            v      v      v
chunk 0 --16KB   chunk 1  chunk 2       <- the actual data

To find row N, you do two cheap operations: divide to pick the chunk (N / rows-per-chunk), and take the remainder to pick the slot inside it (N % rows-per-chunk). Because rows-per-chunk is a power of two, both are just bit shifts.

3.4 The entity handle

An EntityHandle is not a pointer. It’s a small value: an index plus a generation.

  • The index says which slot in a big table describes this entity — that table entry (the “record”) says which archetype the entity is in and which row.
  • The generation is a counter that lets us detect stale handles. When an entity is destroyed and its index is later reused by a different entity, the generation is bumped. An old handle still carries the old generation, so a liveness check (does the handle's generation still match the table's?) fails — the old handle correctly reports “dead,” even though its index now belongs to someone else.

This index→record table is the entity handle table. It’s central to everything that follows.

3.5 The one property that matters: stable addressing

Here is the concept the entire redesign hinges on.

Stable addressing means: once something has been given a memory address, that address never changes for the lifetime of the object — even as the container around it grows.

A plain growable array (std::vector) does not have this property. When it runs out of capacity, it allocates a bigger buffer, copies everything over, and frees the old buffer. Every pointer into the old buffer is now dangling. In single-threaded code that’s fine, because nobody is looking at the old buffer at the moment of the copy. In multithreaded code, it is a catastrophe — which is exactly the problem we hit.

4. Why naive concurrency breaks

Suppose we keep the storage as-is (growable arrays) and just let two threads spawn at once. What goes wrong?

Thread A is filling in a freshly created entity — writing its Position into a column. To do that, it reads the column’s spine to find the chunk, then writes into the chunk. Meanwhile Thread B spawns an entity too, the column needs to grow, and its spine (a growable array of chunk pointers) reallocates — the old spine buffer is freed. Thread A was mid-read of that exact buffer.

sequenceDiagram
    participant A as Thread A (filling row)
    participant S as Column spine (growable array)
    participant B as Thread B (growing column)
    A->>S: read spine to find my chunk
    B->>S: append chunk → REALLOCATE, free old spine
    A->>S: ...continues reading freed memory 💥

This is not theoretical. It crashed in testing with a segfault inside a memcpy, and it is the seed from which the whole design grew. The lesson: for concurrent creation to be safe, the storage that spawners and readers touch must never move.

Three separate things all need that guarantee:

  1. The entity handle table — allocating an index must be safe from any thread, and a record, once addressed, must not move.
  2. The archetype’s columns and EntityHandles — appending a row must not relocate existing rows or the pointers addressing them.
  3. Everything hanging off a spawn — observer notifications, sparse components, slot hierarchies — must each be safe to touch concurrently.

The rest of the paper is those three problems and their solutions.

5. Building block: the lock-free entity handle table

The first structure we made concurrency-safe was the entity handle table. Its job: hand out unique entity indices to any thread, store each entity’s record (archetype + row), and never move a record once it exists.

The design mirrors a proven pattern already in the engine (the object manager), with one addition Catalyst needs.

5.1 A fixed spine that never reallocates

The trick to stable addressing is to make the spine fixed-size and inline. The table holds an array of exactly 1024 chunk pointers, allocated as part of the table itself — it is never resized, so it never moves. Each chunk holds 4096 entity slots. 1024 × 4096 ≈ 4.19 million entities per library.

Growth only ever means: install a chunk pointer into an empty spine slot. It never means “resize the spine.” Because the spine is fixed, a thread that already holds an entity index can compute that entity’s slot address and trust it forever.

Chunks[1024]   (fixed inline array — never moves)
   [0] --> Chunk (4096 slots)
   [1] --> Chunk (4096 slots)
   [2] --> nullptr   <- not allocated yet
   ...
   [1023] --> nullptr

To find a slot from an index: the top bits pick the chunk, the low 12 bits pick the slot within it. Two shifts and a load.

5.2 Compare-and-swap: growing without a lock

When a thread needs a chunk that doesn’t exist yet, how does it add one without a lock — and without two threads both allocating the same chunk?

The answer is an atomic compare-and-swap (CAS). A CAS says, atomically: “if this slot still holds the value I expect (nullptr), replace it with my new value; otherwise tell me it changed.” Two threads racing to install chunk 5 both allocate a candidate chunk and both attempt the CAS. Exactly one wins. The loser sees the CAS fail, throws away its candidate, and adopts the winner’s chunk. No lock, no duplicate, no lost update.

This “allocate speculatively, CAS, free-on-loss” idiom appears everywhere in this project. It is the standard way to grow a shared structure without a lock.

5.3 Recycling indices: the free-list

When an entity is destroyed, its index shouldn’t be wasted. It goes onto a free-list — a queue of indices available for reuse. New allocations prefer to pull from the free-list before extending into fresh territory.

The free-list is a moodycamel::ConcurrentQueue, a well-tested lock-free multi-producer/multi-consumer queue. Producers (threads freeing indices) and consumers (threads allocating) never block each other.

When an index is recycled, its generation is bumped — this is the mechanism from §3.4 that stales old handles. The generation lives in the slot as an atomic, and the bump is paired with the queue’s release/acquire ordering so that a thread which dequeues a recycled index is guaranteed to see the new generation.

5.4 The addition ObjectManager lacks: bulk reservation

A single Allocate() costs one atomic increment of the “next fresh index” counter. Fine for one entity. Terrible for a million: a million atomic increments, all contending on one counter.

So the table has AllocateRange(N): reserve N contiguous indices with one atomic fetch_add(N). A million-entity load pays a handful of atomics instead of a million. This is the path bulk spawning uses.

graph LR
    A["AllocateRange(1,000,000)"] --> B["one fetch_add(1,000,000)"]
    B --> C["indices [start .. start+1,000,000)"]

5.5 A small but real optimization: skip the lookup for fresh indices

A freshly reserved index has never been freed, so its generation is definitionally 0. We originally read the generation out of the slot for every handle we built — a needless two-step lookup through the chunk spine, a million times over during a load. Recognizing that fresh handles are always generation 0 let us skip the lookup entirely and stamp 0 directly. Small idea, measurable on the bulk path.

6. The heart of it: stable-address archetype storage

The handle table was the warm-up. The real work was making an archetype’s columns and EntityHandles stable-addressed, because that is what spawners write into and readers read from.

6.1 The problem with the fixed-spine approach here

You might ask: why not use the same fixed 1024-pointer spine we used for the handle table?

Because the handle table is one structure per library, but there are many columns — one per component type per archetype, across potentially hundreds of archetypes. A fixed spine large enough for a huge archetype (tens of thousands of chunk pointers) would be paid inline, per column, always, even for an archetype holding three entities. The memory blows up. We measured this tension and rejected the fixed spine for columns.

6.2 The solution: an inline-first, paged spine

Instead we made the spine paged, and put a small inline fast path in front of it. A column resolves a chunk’s address one of two ways:

  • Inline chunks. The first InlineChunkCount chunk pointers live directly inside the column, as a tiny inline array of atomics. A column that only ever holds a handful of chunks — the overwhelmingly common case, and every column of a narrow archetype — addresses its data straight out of these, and allocates no spine at all.
  • Overflow spine. Only when a column grows past its inline chunks does it lazily allocate the two-level spine: a fixed inline array of spine-block pointers, each spine-block a page of chunk-pointers, allocated on demand.
graph TD
    IC["InlineChunks[4]<br/>(inline atomics, ~32B — the only cost for a small column)"]
    IC -->|chunk 0| D0["16KB data chunk"]
    IC -->|chunk 1| D1["16KB data chunk"]
    IC -.->|grows past 4 chunks| SP["Overflow spine (lazy)"]
    SP --> SB["SpineBlocks[256]<br/>(allocated only on overflow)"]
    SB -->|block 0| B0["Block: 256 chunk-pointers"]
    B0 -->|ptr 0| C0["16KB data chunk"]

Both levels grow lazily by CAS, exactly like §5.2, and nothing ever moves once installed — not a data chunk, not a spine-block, not the inline array. That is the stable-addressing guarantee. The always-paid cost is now just the inline pointers (a few dozen bytes per column); everything heavier is allocated only when a column actually grows large enough to need it.

Why the inline fast path exists — a measured blow-up. A two-level spine on its own still carried a ~2KB inline top array per column, plus a ~2KB first spine-block the moment a column held any data. §6.1 warned that a fixed per-column spine makes memory blow up for narrow archetypes — and a couple of kilobytes, paid per column, does exactly that once you have many columns. A stress spec that adds 700 distinctly-named slots to a single entity walks it through archetypes of width 1, 2, … 700, creating ~245,000 columns; at ~4KB of pure addressing each, that summed to ~1GB. It was also slow, because building each wide archetype and reserving each row did O(width) spine and block allocations. Moving the common case inline collapsed it: the same spec dropped from ~1190MB to ~280MB and from ~650ms to ~200ms, with no change to the concurrency guarantees. It is the §6.1 lesson applied one level deeper — don’t pay for capacity a column doesn’t use.

Tuning InlineChunkCount. It defaults to 4: four inline chunk pointers, ~32 bytes paid on every column of every archetype, covering any column that stays within four chunks before it promotes to the overflow spine. Because that tax is universal, it is deliberately small. Raise it only if profiling shows a workload with many archetypes that each sit just above the inline count and are churning spine promotions — the trade is a few more always-paid bytes per column against fewer lazy spine allocations for those mid-sized archetypes. The opposite shape — a handful of archetypes each holding millions of entities — is indifferent to the value: those columns promote to the overflow spine almost immediately and amortize it over millions of rows, so leave it at the default. When unsure, keep it at 4; it is sized for the narrow-archetype case, which is where the memory actually lives.

The cost of the paged shape is one extra pointer hop to resolve an overflow chunk’s address (an inline chunk is a single load). We hide it by resolving a chunk’s base address once per chunk during iteration and striding within it, so the extra hop is amortized across hundreds of rows rather than paid per row.

EntityHandles — the per-row handle array — got the identical treatment: it’s a paged store using the same machinery (inline fast path included), configured so a row lands in the same chunk-and-slot as it does in every component column.

6.3 The row count becomes an atomic

Previously “how many rows does this archetype have?” was answered by EntityHandles.size(). With a paged store there is no .size(), and more importantly, the count needs to be claimable atomically. So the archetype now has an atomic RowCount.

Appending a row is now genuinely lock-free:

  1. fetch_add(1) on RowCount — this atomically claims a unique row number. Two threads appending get two different rows, guaranteed, with no lock.
  2. Ensure the one chunk that row lands in exists (CAS growth if needed).
  3. Write the row.

That’s it. Two threads spawning into the same archetype now proceed in parallel: they claim different rows, touch different chunks (or the same chunk at different offsets), and never contend on a lock.

A cautionary tale. The first version of “ensure the chunk exists” re-checked every chunk from zero on every single append. That’s O(rows) work per append, i.e. O(rows²) to build an archetype. A single-entity spawn benchmark went from 23 ms to 669 ms. The fix was obvious once seen — only ensure the chunk the new row actually lands in — but it’s a good reminder that “correct” and “not accidentally quadratic” are different properties, and only benchmarking catches the difference.

6.4 What this bought, and what it cost

Bought: concurrent readers can no longer crash (nothing moves), and same-archetype spawns parallelize. Cost: a single spawn is ~25% slower than the old flat-array version, because an atomic fetch_add and a two-level pointer walk are inherently more work than a bump-and-write into a flat array. We consider that an acceptable price for the safety and parallelism — and §9 explains why we chased, but did not fully eliminate, that 25%.

7. Everything else that hangs off a spawn

Creating an entity isn’t just “append a row.” It also fires observer notifications, may add sparse components, and may set up slot hierarchies. Each needed its own concurrency story.

7.1 Observers: defer, then drain

An observer is a callback that fires when a component is added, removed, etc. Observers run user code, which is not thread-safe, and they read a shared map that isn’t safe to write from many threads.

So during a concurrent spawn, we don’t run observers and we don’t touch the map. We record each notification into a lock-free queue and move on. Later, at a single safe point on the game thread (§8), we drain the queue into the map, and observers run normally. The observers themselves never learn they were fed from multiple threads.

7.2 The finding that mattered most: batch the notifications

This is the most important performance lesson in the whole project, so it gets its own spotlight.

When we first parallelized bulk spawning, it barely got faster — roughly the same as the single-threaded path, despite using every core. We had assumed the bottleneck was a lock. It wasn’t. Careful profiling (with Tracy) showed the culprit was the observer queue itself: recording one notification per entity per component meant a 100,000-entity, 3-component spawn did 300,000 individual lock-free enqueues, and that was eating roughly half of the per-entity spawn time.

The fix was to stop enqueuing per entity. Every entity in a bulk spawn gets the same set of components, so we record one batched notification per component type that carries the whole list of entities. 300,000 enqueues became 3.

Before:  for each entity:  for each component:  enqueue(one entity)     → 300,000 enqueues
After:   for each component: enqueue(all 100,000 entities at once)       → 3 enqueues

That single change is what turned “no faster” into “2.3× faster.” The lock we’d been worried about was never the problem. Measure before you optimize is a cliché precisely because it keeps being true.

7.3 Sparse components and slots

Some component types are stored sparsely — not in the archetype’s columns, but in a side table keyed by entity, for components that only a few entities have. Slot hierarchies (parent/child relationships between an entity’s sub-parts) are implemented on top of this sparse machinery.

Two things made these concurrency-safe:

  • The sparse side-table’s data was made paged (same idea as §6.2), so a pointer it hands back stays valid across later concurrent additions.
  • Each sparse store carries a small mutex covering its bookkeeping (inserting into its map, growing its pages). This is a lock, but a narrow one, and different component types have independent locks.

The one constraint: the set of sparse stores a prefab needs is created when the prefab is baked, not during the spawn. So a worker-thread spawn only ever finds an existing store and adds to it (safe) — it never has to create one (which would mutate the shared store map). This is why prefab baking is currently game-thread-only; see Limitations.

8. The frame model: when is concurrent creation legal?

Concurrency needs a rule for when it’s allowed, or every read everywhere would need to defend against every write. Catalyst’s rule is simple and is expressed as two non-overlapping regions of a frame:

graph LR
    M["MUTATION WINDOW<br/>spawns legal (any thread)<br/>processors NOT running"] --> B{{"BARRIER"}}
    B --> P["PROCESSOR WINDOW<br/>processors run<br/>no structural mutation"]
    P --> M2["next frame's<br/>MUTATION WINDOW"]
  • In the mutation window, any thread may spawn.
  • In the processor window, the engine runs the systems that iterate entities. No structural mutation happens here (systems that want to change structure use the deferred command buffer, unchanged from before).
  • The barrier between them is the safe point. It does two things: it waits for every in-flight spawn to finish, and it drains the observer queue into the map. Only then do processors run.

“Waits for in-flight spawns” is implemented with a counter (ActiveSpawnCount) that each spawn bumps up while it works and drops when done. The barrier spins help-executes on other pending jobs rather than idling — until the counter hits zero. Because a spawn’s final act (dropping the counter) is ordered after all its writes, the barrier seeing zero guarantees every spawned entity is fully formed and visible before any processor reads it.

8.1 One API, main-thread aware

There is a single public Instantiate call. It doesn’t ask the caller which thread they’re on. Internally it checks a cheap thread-local flag: on the game thread it writes observer notifications straight to the map (no queue needed) and skips the spawn-counter bracket; on any other thread it takes the deferred/queued path. Callers never think about it.

9. Single vs. bulk creation

The two spawn paths share machinery but optimize for different shapes.

Single spawn (Instantiate(prefab)): claim one row, fill its columns, publish it. Optimized for latency. We spent real effort shaving its cost: fusing the index allocation with grabbing its record pointer (one lookup through the table instead of two), skipping the generation read for fresh indices (§5.5), and reusing the chunk address that “ensure the chunk exists” already computed instead of looking it up again. These took single-spawn from a regressed ~237 ns down to ~206 ns — still above the old flat-array 165 ns, because, again, the atomic claim and paged walk are irreducibly more work than a flat bump.

Not every idea paid off, and we kept the honest record. We added an approximate free-list-empty counter so allocation could skip a queue probe when there’s nothing to recycle. It’s correct and low-risk, but in our Debug benchmarks it made no measurable difference — the queue probe wasn’t as expensive as we’d guessed. It stays because it can only help under real recycling load, but we flagged it for removal if a profiled build ever shows it’s dead weight.

Bulk spawn (Instantiate(prefab, count, out)): reserve count rows with one atomic (§6.3), then fan the per-entity fill across the worker pool. The batch size handed to each worker is computed from the L1 cache size and the entity’s byte size, the same heuristic the parallel query iterator uses, so each job works on a cache-friendly slice. Above a threshold (1024 entities) it goes parallel; below, the job-dispatch overhead isn’t worth it and it fills inline. This path, with the observer batching from §7.2, is the one that’s 2.3× faster than the original serial code.

10. Reading while the world is being built

The last piece is subtle and worth its own section, because it’s the part most likely to surprise a user.

Catalyst allows ForEach (iterating entities matching a query) to be called anywhere, not just inside a system during the processor window. Editor panels, UI widgets, and tools all do it. We call these out-of-tick reads. And they can happen on a different thread while spawns are running — so an out-of-tick read can overlap the mutation window.

Two guarantees were needed.

10.1 No crashes (came for free)

Because storage is now stable-addressed (§6), an out-of-tick read cannot crash from the reallocation problem of §4. The memory it walks never moves. ForEach snapshots the row count once per archetype at the start, iterates that many rows, and every chunk it touches is stable. This guarantee fell out of the storage redesign at no extra cost.

10.2 No half-built entities (the publication pattern)

Crash-safety isn’t enough. A spawn claims its row (bumping the row count) before it fills in the data. A reader that snapshots the count at that instant would iterate a row that’s been claimed but not yet written — reading uninitialized garbage.

The fix is a classic publish-then-observe pattern:

  • The spawn is split into reserve (claim the row, but leave its handle slot as a null sentinel) and publish (write the handle last, after all the entity’s data is in place, with “release” memory ordering that guarantees the data writes are visible first).
  • Handle chunks are zero-filled on allocation, so a reserved-but-unpublished row reads a null handle — a clear “not ready” marker.
  • The reader skips any row whose handle is null, and uses “acquire” memory ordering so that once it sees a valid handle, the entity’s data is guaranteed visible.
sequenceDiagram
    participant W as Spawner
    participant Slot as Row
    participant R as Out-of-tick reader
    W->>Slot: reserve row (handle = null)
    W->>Slot: write component data
    W->>Slot: publish handle (release) ✅
    Note over R: reads handle (acquire)
    R->>Slot: null? → skip.  valid? → data is safe to read

There’s a wrinkle we handled: if a row was previously used, destroyed, and is now being re-reserved, its slot still holds the old handle — not null. So the destroy path nulls the vacated slot, preserving the invariant “a reserved-but-unpublished row always reads null.”

10.3 Paying nothing when nobody’s spawning

The per-row null check isn’t free, and out-of-tick reads are common — so charging every out-of-tick read for a safety check that only matters during a spawn would be a bad trade. The first implementation did exactly that and regressed iteration from 6.9 ns/row to 18.5.

The fix: the check runs only when out-of-tick AND a spawn is actually in flight (ActiveSpawnCount != 0), read after the row-count snapshot so that seeing zero provably means every counted row is published. The overwhelmingly common case — an out-of-tick read with no concurrent spawn — takes the identical fast path as in-tick iteration and pays nothing. Measured back at 6.9 ns/row, unchanged from baseline.

This is validated by a stress test that runs 100,000 concurrent spawns while a reader thread hammers ForEach in a loop: it never observes a half-built entity and never crashes.

11. Current limitations

We are deliberately honest about what this does not do.

  • Single-spawn is ~25% slower than the pre-concurrency flat-array path (~206 ns vs ~165 ns), but only in debug config. Once you use release config, it actually flips - single-spawn becomes about 25% faster.
  • Prefab baking is still game-thread-only. Baking creates archetypes and sparse stores (mutating shared maps). Instantiating a baked prefab is fully concurrent; baking one is not. In practice you bake at load time and instantiate at runtime, so this rarely bites — but it’s a real constraint, tracked as future work.
  • Sparse-filtered reads during spawns have a visibility nuance. An out-of-tick read that filters on a sparse component can momentarily fail to match an entity whose archetype handle is published but whose sparse element is still mid-insertion. It skips that entity (never reads garbage) — the entity simply appears one iteration late. Harmless in practice; tightenable if it ever matters.
  • ~4.19 million entities per library. The handle table’s fixed spine caps a single library at 1024 × 4096 slots. Raising it is a one-constant change; we just haven’t needed to.
  • Memory overhead. Stable addressing isn’t free in memory: each column and the handle store carry a small always-paid inline chunk array (InlineChunkCount pointers, ~32 bytes at the default of 4), where the old flat array started at nearly nothing. This is deliberately tiny — an earlier design paid a ~2KB spine per column and blew up to ~1GB under archetype explosion (§6.2) — but it is still a per-column tax, so an archetype with many components and few entities pays a few dozen bytes per column it barely uses.
  • No concurrent destruction. By design (§2). Removing entities from multiple threads is not supported; use entity disabling.

12. Future work

  • Concurrent destruction. The hard part of deletion is that removing a row usually means moving the last row into the hole (a “swap-remove”), which relocates a live entity — the exact “moving data” hazard we avoided for creation. The plausible design is tombstoning: a concurrent “delete” only marks the row dead (an atomic write of the null-handle sentinel — the very check §10.2 already performs), and the actual physical compaction happens later at the barrier when no reader or spawner is live. It’s a natural extension, not a rewrite, but it wasn’t a priority.
  • Thread-local index blocks (a “TLAB”). Borrowed from allocator/GC terminology: instead of every thread hitting the shared index counter for one index at a time, each thread grabs a block of indices in one atomic and hands them out privately with no further atomics until the block runs dry. It would help the “many threads each spawning singles, heavily” pattern. Deferred because bulk already has its own one-atomic path, and single-spawn atomic contention hasn’t profiled as a problem. Its wrinkle is wasted indices when a block is only partly used.
  • Cheaper bulk chunk allocation. After the observer fix, the dominant remaining bulk cost is simply allocating the chunks, and it does not parallelize. A bulk spawn of 100 K entities allocates on the order of 3,000 small new[] chunks (the count grows with both chunk count and column count, because more components per entity means fewer entities per 16 KB chunk and more columns to allocate). Profiled on a 32-thread machine, this shows up as a serial ~2.3 ms prologue in AddEntities for a 3-component spawn — pure Amdahl serial fraction that no amount of fill parallelism can move. We tried the obvious fix — allocate lazily inside the parallel fill, so each worker new[]s the chunks its own row-range enters. This failed, and instructively so: it did not remove the serialization, it relocated it into the heap allocator’s internal lock. The default Windows heap serializes concurrent allocations, so 32 workers hitting new[] at their chunk boundaries simultaneously pile up on that lock — whichever worker holds it shows a ~1.8 ms spike inside a single “spawn entity” zone, and the total comes out slightly worse than the serial version. The lesson: the problem was never where the allocation happens, it’s the sheer count of small allocations; no allocator, parallel or serial, scales to thousands of tiny new[]s per operation. The viable fix is therefore to collapse the allocation count, not distribute it: for a bulk spawn of known size, allocate one large slab per column covering the whole contiguous row-range (turning ~3,000 allocations into ~4) and point the spine chunk entries into it. The serial reserve then becomes a handful of large VirtualAllocs, and page-faulting on first touch stays in the parallel fill where it scales. The real cost is on the free side: today each spine pointer is independently delete[]-able, so slab-backing requires the column to own its slabs (OwnedBlocks) with the spine entries demoted to non-owning views, and the concurrent single-spawn EnsureChunk path to register its one-chunk allocations under a light lock. A contained change to ComponentColumn, but one that touches the free path and the single-spawn hot path, so it is deliberately deferred rather than layered on.
  • Thread-safe baking. Making BakePrefab callable off the game thread removes the last single-thread constraint in the creation pipeline.
  • The job-scheduler question. Profiling surfaced occasional ~200 µs gaps at the start/end of parallel-fill jobs, and per-entity cost spikes that look like cache effects. These point at the job system and the cache-size batching heuristic rather than Catalyst storage, and are worth a separate investigation.

13. Conclusion

Concurrent entity creation in Catalyst came down to a single principle — stable addressing — applied uniformly: to the entity handle table, to archetype columns, and to the per-row handle array. Once memory never moves, appends become lock-free atomic row claims, and reads become safe by construction. Layered on top are a deferred-and-batched observer path, narrow locks for the sparse side-tables, a frame barrier that fences the mutation window from the processor window, and a publish-then-observe protocol that lets reads run concurrently with spawns without ever seeing a half-built entity.

The engineering lessons generalize beyond this feature:

  1. Stable addressing is the enabling property for lock-free concurrency over growable data. Get that, and most of the hard parts dissolve.
  2. Your assumed bottleneck is probably wrong. We spent effort removing a lock that was never the problem; the real cost was per-item queue traffic, invisible until profiled and obvious afterward.
  3. “Correct” and “fast” are separate checks. A quadratic loop and a per-row safety check both passed correctness and both quietly cost an order of magnitude until measured.

The result: any thread can create an entity and use it immediately.

If you are making your own game engine, or working in an existing engine and are asking yourself: Is it worthwhile to do this? Honestly, for performance, it’s not. Spawning entities is already blazing fast. There’s simply too many moving parts in a fully multithreadable ECS data storage which just cripples its performance.

Here is a table from a benchmark that would spawn 100k entities on Alchemy’s release config back when it was fully serial, no multithreading:

6.7 milliseconds for 100k entities, and that’s the inefficient method. Bulk goes down to 2.1, and a complicated entity goes up to 3.8.

Here is the same table, but after the concurrent spawning changes:

Here’s the thing, the performance difference is actually far worse than you’d think. You would think that on my machine, a 9950x with 32 threads that this bulk parallel spawning should be at minimum 3x faster. But no, it’s actually at best 25% faster, and at worst equal. But remember, the second table has 32 threads. The first one has one. The second table should be running circles around the first table. The good news is: §12 (Future work) has a ton of solutions to fix this horrible performance scaling. When we look at these two tables inside a Tracy file, the bulk instantiate in the serial version was taking 1.9ms. Concurrent version was 800 μs, so about twice as fast (and that’s including the odd gaps in the job distribution), but using all 32 threads. Preparing all the chunks, handles, etc, is what eats up those 3ms and makes us match the old serial version.

Most of the world comprises of entities that only have a handful of components, like a transform and mesh component so it’s not something we’re really worried about. Most of the time that you are spawning this many entities it’s during a loading screen where you can hide most of these spikes. And even then, you can implement a frame-budgeted spawn system. You start a job that spawns X entities in Y amount of time, and only give it Z amount of time per frame.

So why did Alchemy do this? Alchemy did it primarily because of how multithreaded the entire engine is. World streaming involves spawning many entities on multiple threads, the asset registry involves spawning many entities on multiple threads, and many other sections needed to spawn entities on any thread. For Alchemy, this was actually necessary. But this was one of those rare circumstances where going serial to parallel actually harmed performance (the entire CPU is now required with this system to match the speed of the single thread version). Hopefully after the points found in the future-work section is tackled, we will finally have some proper numbers to be proud of.

If you are making a single game, maybe not even with a game editor, I would say; don’t bother with concurrent entity spawning. It’s a lot of work, with a lot of high-risk choices. There’s a reason why you don’t see this feature in big ECS frameworks.
