Build System

Alchemy Engine uses a two-stage build system: ABT (Alchemy Build Tool) generates module metadata, and CMake compiles the code.

Overview

GenerateProjectFiles.bat
  |
  +-- Step 1: ABT.exe (C# tool)
  |     - Scans Source/ directories for .build.cs files
  |     - Parses module configuration (dependencies, definitions, libraries)
  |     - Generates root CMakeLists.txt with all module targets
  |
  +-- Step 2: CMake
        - Reads CMakeLists.txt (root, generated by ABT)
        - Includes Source/ThirdParty/ (third-party library targets)
        - All engine/runtime/application/editor/plugin modules defined inline
        - Generates Visual Studio solution at Build/AlchemyEngine.sln

ABT (Alchemy Build Tool)

ABT is a .NET 8 console application located at Tools/ABT/. It discovers modules and generates a monolithic CMakeLists.txt containing all module targets.

Location

  • Source: Tools/ABT/*.cs
  • Project file: Tools/ABT/AlchemyBuildTool.csproj
  • Built executable: Tools/ABT/bin/Release/net8.0/AlchemyBuildTool.exe

Building ABT

How ABT Works

  1. Module Discovery (ModuleDiscoverer.cs): Scans these directories for .build.cs files: - Source/Engine/ - Core engine modules (type: EngineCore)
  • Source/Runtime/ - Runtime modules (type: Runtime)
  • Source/Applications/ - Executable targets (type: Application)
  • Plugins/ - Engine-bundled plugins (type: EnginePlugin)
  • {ProjectDir}/Source/ - Project modules (type: ProjectModule)
  • {ProjectDir}/Plugins/ - Project plugins (type: ProjectPlugin)
  1. Build Rules Loading (BuildRulesLoader.cs): Parses each .build.cs file using regex to extract: - PublicDependencyModuleNames.Add("...")
  • PrivateDependencyModuleNames.Add("...")
  • ThirdPartyLibraries.Add("...")
  • PublicDefinitions.Add("...")
  • PrivateDefinitions.Add("...")
  • PublicIncludePaths.Add("...")
  • IsSharedLibrary = false
  1. CMake Generation (CMakeGenerator.cs): Writes root CMakeLists.txt with all module target definitions inline (no per-module CMakeLists.txt files).

ABT Source Files

FilePurpose
Program.csEntry point. Orchestrates discovery, loading, and generation.

.build.cs Files

Each module has a .build.cs file that declares its build configuration. The file name determines the module name (e.g., AlchemyEngineCore.build.cs defines the AlchemyEngineCore module).

Example: AlchemyEngineCore.build.cs

Available Properties

PropertyScopeDescription

Plugin .build.cs Files

Plugin modules have the same .build.cs format as engine modules. An empty file is valid — the build system automatically applies the correct engine include paths and links AlchemyEngineCore for all plugin targets.

If a plugin needs additional third-party libraries:

Plugin modules are always built as shared libraries (DLLs). Setting IsSharedLibrary in a plugin .build.cs has no effect — plugins always compile to SHARED.

Documentation Linking

Modules can embed markdown documentation directly alongside source code in the solution explorer using DocumentationPaths. This makes documentation browsable in the IDE without leaving the project.

Syntax:

ArgumentDescription
TargetFolderThe folder name in the solution explorer where the docs appear (e.g., "Catalyst" places them under the Catalyst code folder)
SourcePathPath to a markdown file or directory, relative to the engine root
DisplayNameOptional. Subfolder name under TargetFolder. When omitted, docs go directly into TargetFolder

Example:

This produces the following hierarchy in the solution explorer:

AlchemyEngineCore/
  Catalyst/
    Archetype.h
    Entity.h
    ...
    Documentation/        <-- from DocumentationPaths
      Catalyst.md
      Components.md
      Entity Query.md
      ...

Implementation note: ABT copies referenced documentation files into Build/{ModuleRelPath}/ rather than referencing them in-place from Documentation/. This is a workaround for JetBrains Rider not fully supporting .vcxproj.filters for CMake-generated projects — Rider resolves files by filesystem path, so the copies must mirror the source tree structure to merge correctly. If Rider adds full .vcxproj.filters support in a future release (tracked on their roadmap), this copy step could be replaced with direct references. Visual Studio reads the filters correctly either way.

CMake Structure

Root CMakeLists.txt

The root CMakeLists.txt is fully generated by ABT and contains all module targets inline. Do not edit it manually — regenerate with GenerateProjectFiles.bat.

It does the following:

  1. Sets C++23 standard, MSVC compiler options, build configurations
  2. Finds Python (for Reflector.py) and Vulkan SDK
  3. Runs the reflection system (Tools/Reflector.py)
  4. Sets global include directories (Source/, ThirdParty headers)
  5. Adds Source/ThirdParty via add_subdirectory()
  6. Defines all module targets (engine, runtime, editor, application, plugin) with their sources, dependencies, and definitions
  7. Copies documentation files into Build/ for IDE integration
  8. Sets up the Documentation virtual folder and VS startup project

Third-Party Libraries

All third-party libraries are defined in Source/ThirdParty/CMakeLists.txt. Each library is a CMake target with its own target_include_directories() set to PUBLIC or INTERFACE. When a module links to a third-party library via target_link_libraries(), the include directories propagate automatically through CMake’s transitive dependency system.

GenerateProjectFiles.bat

The main entry point for project generation.

What It Does

  1. Sets up Visual Studio environment (vcvars64.bat)
  2. Finds the Vulkan SDK
  3. Backs up Build/.idea/ (Rider settings including git config)
  4. Deletes and recreates Build/
  5. Restores Build/.idea/
  6. Runs ABT to generate root CMakeLists.txt
  7. Runs CMake to generate Visual Studio solution at Build/AlchemyEngine.sln

When to Run

  • After cloning the repository
  • After adding/removing source files
  • After modifying any .build.cs file
  • After modifying CMakeLists.txt
  • After running Setup.bat (which clones third-party repos)

Build Configurations

ConfigurationDefinesDescription
DebugDEBUG_BUILD=1, ALCHEMY_WITH_CRUCIBLE=1Full debugging, tests enabled
RelWithDebInfoDEVELOPMENT_BUILD=1, ALCHEMY_WITH_CRUCIBLE=1Optimized with debug info, tests enabled
ReleaseRELEASE_BUILD=1Fully optimized, no tests

Output

Build artifacts go to Build/bin/{Config}/:

  • AlchemyEngineCore.dll - Core engine
  • AlchemyRuntime.dll - Runtime module (hot-reloadable)
  • AlchemyFrontend.exe - Editor application
  • AlchemyRunner.exe - Project runner

Plugin System

Plugins are directories containing a .plugin JSON manifest and a Modules/ subdirectory with .build.cs files:

Plugins/
  MyPlugin/
    MyPlugin.plugin        <- JSON manifest
    Modules/
      MyPlugin/
        MyPlugin.build.cs
        *.cpp / *.h
    Binaries/              <- DLL output location

ABT discovers plugins automatically. Each plugin module compiles to its own DLL, output to the plugin’s Binaries/ directory.

Per-Module Hot Reload

The current hot-reload system watches a single DLL (AlchemyRuntime.dll). This involves a ModuleManager that tracks multiple DLLs and reloads them individually based on the dependency graph from ABT.
