Object System

The Object System is the foundational memory and referencing architecture of the Alchemy Engine. Unlike older monolithic engines that rely on global garbage collectors, Alchemy uses a highly optimized, lock-free Intrusive Reference Counting and Handle-Based Weak Referencing system.

This allows thousands of threads to allocate, reference, and destroy objects simultaneously with zero mutex contention and zero garbage collection hitches.

The Base Object

Object is the fundamental base class for game logic elements. It operates at exactly 48 bytes and contains:

  • vptr (8 bytes)
  • Owner (8 bytes)
  • ObjectHandle (8 bytes)
  • Guid (16 bytes)
  • RefCount (4 bytes)
  • ObjectFlags (4 bytes)

Objects are entirely responsible for their own lifetime via their RefCount. When the reference count drops to 0, the object destroys itself instantaneously.

Pointer Types

The engine provides three distinct smart pointers to interact with objects safely. Never use raw pointers for object ownership.

1. ObjectPtr<T> (Strong Pointer)

The standard strong pointer. Holding an ObjectPtr increments the internal RefCount of the object, mathematically guaranteeing it cannot be destroyed while you hold the pointer.

  • Cache Penalty: 0 (Direct raw pointer access)
  • Performance: Extremely fast. Use this in tight loops and performance-critical code.
  • Rules: You must be careful to avoid cyclic references (e.g., Object A holds a strong pointer to Object B, and Object B holds a strong pointer to Object A), which will cause memory leaks.

2. WeakObjectPtr<T> (Handle Pointer)

A weak pointer that does not increment the reference count. If the object dies, the weak pointer will safely return nullptr instead of crashing.

  • How it works: It holds an 8-byte ObjectHandle (32-bit Index, 32-bit Generation). When you call Get(), it looks up the flat array in the ObjectManager to ensure the object hasn’t been destroyed and the slot hasn’t been reused.
  • Performance: Requires an O(1) array lookup and generation check. Extremely fast, but not “free”.
  • Usage: Use for high-level gameplay logic (e.g., tracking a target, UI references, parent/child safety links).

3. SoftAssetPtr<T> (Asset Reference)

Used strictly for referencing assets on disk (e.g., /Content/Meshes/Player.alchemyasset).

  • It holds a string path to an asset that may or may not be loaded in memory.
  • It integrates with the asynchronous streaming system to load assets on demand.

The ObjectManager

The ObjectManager is a global singleton responsible for routing ObjectHandle lookups for Weak Pointers.

To ensure maximum multi-threading performance, it is completely Lock-Free.

  1. Chunked Array: It uses an array of pointers to memory blocks (Chunks of 65,536 slots). This prevents the engine from ever having to resize or copy massive arrays, eliminating lag spikes.
  2. Lock-Free Spawning: Threads claim new array indices using fetch_add or by popping from a moodycamel::ConcurrentQueue recycle bin.
  3. Compare-And-Swap Expansion: If the chunked array needs to expand, threads race to allocate the new chunk using atomic CAS. No mutexes are ever used.

Spawning Objects

Always spawn objects using the NewObject<T> template. This properly initializes the RefCount, generates the serialization GUID, and registers the object with the lock-free ObjectManager to receive its Handle.

Serialization

Objects serialize through ObjectSerializer (Engine/Object/ObjectSerialization.h), which reads and writes whole sets of objects at once. An object persists as its dynamic (most-derived) reflected type identity, its Guid, its persistent flags, its owner (by Guid), and every reflected field it carries. The field data itself comes from the reflection system: each reflected type registers a TypeInfo::Serialize function, and ObjectSerializer invokes it for every level of the object’s inheritance chain, so a field declared on a base class is saved and restored alongside the derived type’s own fields.

References resolve by GUID, never by pointer

Because ObjectPtr and WeakObjectPtr store a raw pointer and a runtime ObjectHandle respectively, neither of which survives a save/load, both serialize as the Guid of the object they target. On save, the pointer writes its target’s Guid (an all-zero Guid for a null pointer). On load, it reads that Guid and asks the Archive’s installed IObjectResolver to turn it back into a live pointer. A weak pointer additionally rebuilds its handle from the resolved object; a strong pointer takes a fresh reference. A Guid the resolver does not recognise — a reference to an object outside the saved set — restores as null rather than dangling.

Loading is ordering-independent

ObjectSerializer writes the identity of every object as one contiguous table, followed by the field data. This lets the loader run two passes over the archive:

  1. The first pass reads the identity table and allocates every object — assigning each its saved Guid, a fresh ObjectManager handle, and an entry in the ObjectResolver — before a single field is read.
  2. The second pass reads the field data with the resolver installed, so any ObjectPtr/WeakObjectPtr field (and the owner link) resolves its stored Guid against objects that are all guaranteed to already exist.

The order objects appear in the saved set is therefore irrelevant. If ObjectA references ObjectB and ObjectA happens to be allocated first, ObjectB still exists by the time ObjectA’s fields are read, because no field is read until the whole set has been allocated. A reference is only ever read during the second pass — the field serializers never dereference a target while allocating — so a not-yet-constructed target can never be observed.

Ownership of the loaded graph

ObjectSerializer::LoadObjects returns the reconstructed objects as an Array<ObjectPtr<Object>>. Those strong pointers own the loaded graph: the objects live exactly as long as the caller keeps the returned pointers (or copies of them), and dropping them all destroys the graph through the normal reference-counting path.
