String

String is Alchemy Engine’s owning string type — the engine’s equivalent of std::string, but built around the access patterns a game engine actually has: lots of short strings (asset paths, identifiers, tags), keys compared over and over, and a strong preference for avoiding allocations.

It is not a thin wrapper. String makes a few deliberate trades — it is slightly larger than std::string — in exchange for small-string optimization, a cached hash for fast repeated comparison, and first-class UTF-8 byte/character bookkeeping.

If you only need to look at a string

Most “read-only” work (slicing, searching, comparing a substring) does not need an owning String at all. Reach for String View, which is a non-owning window and avoids allocation entirely.

Why it exists

String targets three things std::string does not give you out of the box:

  1. Small-string optimization (SSO) that keeps short strings entirely inline — no heap allocation.
  2. A cached hash, so a string used as a key and compared repeatedly costs O(1) per comparison instead of a full byte scan.
  3. Explicit UTF-8 semantics — it always knows both its byte length and its character (code-point) count, and never conflates them.

Small-string optimization

A String stores up to 23 bytes inline (24 including the null terminator). Below that threshold there is no heap allocation at all — construction is just a copy into the object’s own storage. Once a string exceeds the inline capacity it transparently moves to the heap and can grow to any size.

This is why short strings are effectively free to create. In practice the overwhelming majority of engine strings — component names, short identifiers, tags, file stems — never touch the heap.

The practical guidance: keep strings short where you can, and they cost you almost nothing.

Bytes vs. characters

This is the most important thing to understand about String, and the easiest place to introduce a bug if you ignore it.

UTF-8 encodes a character (code point) in one to four bytes. So "Hello, 世界" is 9 characters but 13 bytes (the two CJK characters are 3 bytes each). String tracks both, and exposes them as two clearly separated queries:

You want…UseReturnsUse it for
The raw storage sizeGetByteSize()bytesbuffers, indexing, memory, interop with C APIs
The user-visible lengthGetLength()code pointsUI, cursor positions, “how many characters”

Everything internal works in bytes. Indexing (operator[]), Find, and Substring all use byte offsets, not character offsets — consistent with how the data is actually laid out. GetByteSize() is O(1) (it is stored); GetLength() is computed on demand by walking the bytes, since the character count is rarely needed in hot paths.

Don't assume one byte per character

A common reflex carried over from ASCII is to treat size as both “bytes” and “characters.” For any non-ASCII content these differ. If you are sizing a buffer, you want GetByteSize(). If you are counting what a human sees, you want GetLength().

UTF-8 source literals

String is built to hold UTF-8 bytes. A u8"..." literal is always UTF-8 regardless of compiler settings and is the safest way to embed non-ASCII text. Plain narrow "..." literals also work because the engine compiles with MSVC’s /utf-8 flag (which forces both the source and execution character sets to UTF-8) — without that flag, narrow literals would be interpreted in the machine’s local code page and non-ASCII text would be mangled.

The cached hash

String caches a 64-bit FNV-1a hash of its contents. The cache is lazy: the hash is only computed the first time it is needed (an explicit GetHash(), or a case-sensitive comparison), and a value of 0 is used as the “dirty” sentinel — any mutation resets the cache to 0, so it is recomputed on next use.

The payoff is fast repeated comparison. Case-sensitive equality first compares byte length, then the cached hashes; only if the hashes match does it confirm with an actual byte comparison (to rule out the vanishingly rare 64-bit collision). For two strings compared many times — exactly what happens with map keys, asset IDs, and lookups — every comparison after the first is effectively O(1).

When the cached hash helps, and when it doesn't

  • Helps: long-ish strings compared repeatedly (map keys, IDs). The first compare warms the hash; the rest are O(1).
  • Doesn’t help: very short strings (a few-byte memcmp is already faster than the hash machinery), or a string compared exactly once (you pay a full hash pass and then a compare — slower than just comparing).

The design assumes you “modify rarely, compare often.” If you plan to compare a string a lot, mutate it as little as possible so the cached hash stays warm.

Memory layout

String is 40 bytes, with zero padding. The fields are packed deliberately:

CachedHash   8 bytes   (0 means "dirty / needs recompute")
Capacity     4 bytes   (top bit = "data is on the heap"; low 31 bits = real capacity)
ByteSize     4 bytes   (storage size in bytes, excluding the null terminator)
union        24 bytes  (inline char buffer, OR a heap pointer)

Two pieces of state that naive string types spend whole fields on are folded in: the “hash is dirty” flag is encoded as CachedHash == 0, and the “using heap” flag is the top bit of Capacity (an SSO string never needs a 2-gigabyte capacity, so the bit is free). The heap flag is stored, not inferred from the capacity’s magnitude — so a future change to growth or capacity handling can’t silently misclassify inline vs. heap storage.

For comparison, MSVC’s std::string is 32 bytes. String trades those extra 8 bytes for SSO plus the cached hash. If you are storing hundreds of thousands of strings and memory is critical, prefer String View for the read-only ones.

String View

StringView is a non-owning window into character data it does not control. It stores only a pointer and a byte length, so constructing one and slicing it is effectively free — no allocation, no copy.

Use it whenever you need to read or slice without taking ownership:

Both forms support the same read-only operations — Find, StartsWith, EndsWith, Contains, Equals, byte/character counts.

A view does not own its data

A StringView is only valid while the data it points at stays alive and unchanged. A view obtained from a String is invalidated the moment that String is mutated (a mutation may reallocate) or destroyed. Treat views as short-lived; never store one past the lifetime of its source.

A view is also not guaranteed to be null-terminated — Substring can carve a slice out of the middle of a buffer. Every StringView operation is therefore length-bounded and never relies on a terminator. (String, by contrast, is always null-terminated, so its GetCharData() is safe to hand to C APIs.)

Appending in a loop

Repeated s += x / Append grows capacity geometrically, so building a string piece by piece is amortized O(n). Reserve exists if you know the final size up front and want to skip the intermediate growth.

Gotchas

  • Byte indexing. operator[], Find, and Substring use byte offsets. Indexing into the middle of a multibyte code point gives you a single byte, not a character.
  • Non-const access invalidates the cached hash. A non-const operator[] or begin() hands out writable access, so it conservatively marks the hash dirty (it cannot know whether you will write). Prefer const access in hot read paths so the cached hash stays warm.
  • View lifetime. As above — a StringView does not keep its source alive and is invalidated by any mutation of the source.
  • Case conversion is ASCII-only. ToLower/ToUpper fold ASCII letters and leave multibyte code points untouched. Full Unicode case folding is out of scope.
  • Ordering is byte-lexicographic. Comparison operators sort by raw bytes, which for UTF-8 also matches code-point order — suitable for ordered containers and sorting.
