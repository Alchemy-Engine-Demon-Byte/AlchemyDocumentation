Delegates

A Delegate is Alchemy’s serializable function callback — the replacement for a bare std::function in any type that needs to be saved, loaded, or authored in data. It lets a UI button, an Object, or an entity component hold “call this when clicked” as persistent state rather than as a raw closure that evaporates on serialization.

The core principle

You cannot serialize a callable. A std::function wraps a closure — captured state plus a code pointer that only exists in the running process. There is no stable, cross-build identity to write to disk.

What a Delegate serializes instead is a receiver plus a reflected method name that resolves at load, never the callable itself. A Delegate therefore holds one of two things:

  • A runtime closure — a lambda bound in C++. Ergonomic, captures live state, and is never serialized (it is simply absent after a round-trip). This is the code-authored path.
  • A method binding — a receiver plus a reflected method name. This is what survives save/load and can be authored in data.

The serializer drops the half that cannot cross the disk boundary — a runtime closure writes nothing and reads back unbound.

The two delegate types

A single-cast delegate has one binder, so it may return a value directly. An array (multi-cast) delegate holds many binders, so a scalar return would be ambiguous — instead a non-void signature collects one result per live binder into an Array<Ret>, in bind order.

The signature passed to the macro is a full function type, so parameters and return type are declared exactly as they read in C++.

Binding

Three ways to bind. Only method bindings on an identity-bearing receiver serialize; a closure is runtime-only.

Invocation runs directly through the member pointer — no lookup on the bind-in-code path. The method’s name is captured for serialization via MemberName, so no string is ever typed by hand. The closures re-resolve their receiver on every call (a WeakObjectPtr for an object; a re-fetch from the library for a component), so a destroyed receiver or an archetype migration that moves component storage becomes a safe no-op rather than a dangling call.

&T::Method is ambiguous for overloaded methods, and the reflector gives overloaded names no invoker. Bind non-overloaded methods.

If a binding cannot be resolved on load (the receiver is gone, or the method no longer exists), the delegate reads back unbound. One missing binding never corrupts the rest of a loaded asset — the same graceful-degradation stance the Archive takes toward missing types.

Receiver identity

A method binding’s receiver must have a serializable identity. That constraint defines which receivers can be persisted:

KindIdentity on diskResolved on load via
None—A runtime closure or an unbound delegate. Not serialized.
Objectthe object’s GuidThe Object system’s IObjectResolver, attached to the Archive during an object-graph load.
Entitythe entity handle + component type + slot nameThe EntityLibrary attached to the Archive during library (de)serialization.

A raw C++ struct, a stack object, or a lambda capturing locals has no persistent identity, so it cannot be a serialized receiver — binding to one is runtime-only.

GUID is the disk form; Handle is the runtime form (Objects)

An Object holds both a Guid and a runtime ObjectHandle. An Object binding serializes the receiver’s Guid (through WeakObjectPtr’s serializer) and, on load, resolves it back to the live object through the IObjectResolver the Object deserializer installs on the Archive. Runtime dispatch is Handle-based — no Guid lookup happens per call.

Entity bindings are library-scoped

An EntityHandle is an index + generation into one library’s table; it has no meaning outside that library. So an entity binding persists the handle, the component type, and the slot name, and resolves against the EntityLibrary the Archive carries during library (de)serialization — a whole-library save/load preserves handles exactly, so an intra-library binding round-trips.

Moving an entity to a different library severs its delegate bindings. A cross-library move (merge, or prefab instantiation into another library) reassigns the handle, and only plain reflected EntityHandle fields are remapped by the field-walk — a handle stored inside a delegate is not reached. Such a binding reads back unbound in the new library and must be re-established there. Keep entity references that must survive a move as plain EntityHandle fields.

How name resolution works

Methods are reflected by default, exactly as fields are — reflection is opt-out, not opt-in. Two pieces of reflection metadata make a method bindable by name:

  • Compile-time name. MemberName<&T::Method>() extracts the bare method name at compile time by parsing the compiler’s own signature string (__FUNCSIG__ on MSVC). It costs nothing at runtime and, because &T::Method must compile, a rename breaks the build rather than producing a stale string. This is how a bind captures the name to serialize.
  • Runtime invoker. For every uniquely-named reflected method the reflector emits a type-erased invoker via MakeMethodBinding<&T::Method>(), stored on the method’s MethodInfo alongside a typeid of its call signature. On load, FindMethodInvoker(type, name, signature) returns that invoker — the signature check guarantees it is safe to call — and the delegate rebuilds its closure around it.

The compiler derives the invoker’s exact parameter and return types from the member pointer, so nothing parses a signature as text. A method whose signature the string-parser reads imperfectly still compiles, and an overloaded (or deleted) method simply gets no invoker — it is reflected as metadata only.

Array (multi-cast) delegates

A MulticastDelegate holds a list of binders — the event pattern: one Invoke calls all of them in bind order. Each binder is itself a single-cast Delegate, so the same three binding styles apply, and the serializable ones persist:

Serialization writes only the serializable binders (runtime closures are skipped); on load, a binder that fails to resolve is dropped rather than kept as a dead entry, so Num() and Invoke reflect only the binders that will actually fire.

Not yet implemented

  • Subsystem receivers. A singleton subsystem referenced by type would be a receiver-less-instance target (the DelegateTargetKind enum reserves room for it). Not built yet.
  • Bound payloads. Baking extra arguments into a binding (so one reflected method serves many data-parameterized bindings) via an InstancedStruct is a possible future addition. Its necessity over an exact-signature match is still under review, so it is intentionally not built.
  • Cross-asset late binding. Resolving a receiver that is not part of the same load pass (a delegate loaded before its receiver exists) would need a persistent, lazy resolver rather than the transient load-time context. Out of scope for now.
