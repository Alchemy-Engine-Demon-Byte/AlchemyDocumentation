Input Profiles

An InputProfile owns a set of named InputActions. Each action maps one or more physical inputs (keys, axes, buttons) to a logical name that game code queries.

If you’re coming from Unreal Engine, a good way to think about profiles is that it’s like a Input Mapping Context, but the Input Action’s are defined inside of it. You do not create separate Input Action assets.

Profiles are defined once and shared across entities by pointer. All soldiers, all vehicles, all UI cursors can reference the same profile object.

Defining a Profile

Binding an Entity to a Profile

Add InputReceiverComponent (what device feeds this entity) and one or more InputProfileComponent slots (which profile to apply).

Multiple Profile Slots (Context Switching)

Entities can carry multiple profile slots that can be enabled/disabled via Library.EnableComponent() and Library.DisableComponent(). The slot name is the identifier — it answers “are you in the vehicle context?“.

The router processes all enabled slots and merges their resolved actions into InputStateComponent each frame.

For more information about Component Slots, see Components.

Modifiers

Modifiers are applied in order to the resolved value each frame, after the raw value is read.

ModifierPurpose
DeadZoneModifier(Low, High=1.0f)Ignore values below Low; remap [Low, High] → [0, 1]
SensitivityModifier(Scale)Multiply value or axis by a scale factor
NegateModifierFlip the sign
NormalizeModifierClamp 2D axis to unit circle (fixes diagonal speed)
ResponseCurveModifier(Exponent)Exponential response curve
SmoothModifier(Time)Lerp toward target over time (stateful)

Modifiers are chained. The output of one becomes the input of the next:

Raw value 0.8f
    → DeadZoneModifier(0.1f)     →  0.78f   (remapped)
    → SensitivityModifier(1.5f)  →  1.17f   (scaled)
    → final value: 1.17f
